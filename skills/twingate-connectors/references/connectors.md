# Twingate Connectors Overview

## Summary
Connectors are Twingate components deployed behind your firewall to provide access to private Resources. They run as containers or Linux systemd services and are managed through the Twingate admin console.

## Key Information
- Connectors run as either a **container** or **Linux systemd service**
- Admin console provides ready-made deployment scripts for supported environments
- Connector names are randomly generated at creation but can be edited
- Admins receive email notifications when Connectors go offline/come back online

## Supported Deployment Environments
- Docker
- Kubernetes (via Helm Chart)
- Azure (via ContainerInstance)
- Linux (generic systemd deployment script)
- AWS ECS Fargate
- AWS AMI

## Prerequisites
- Access to Twingate Admin console
- Target deployment environment provisioned
- Firewall access for Connector deployment

## Gotchas
- **Windows/Docker not recommended**: Known Docker issue affects Windows deployments; use Linux VM via Hyper-V instead
- **Connector naming**: Names must be unique across all Connectors in the account
- **Name sync gap**: Renaming a Connector in the Admin console does **not** update the name in your deployment environment — rename before deploying if custom names are needed
- **Email notifications**: Status availability emails are enabled by default; must be manually disabled per-Connector if unwanted

## Configuration Notes
- No environment variables or CLI flags documented on this page
- Deployment scripts are generated by the Admin console per environment
