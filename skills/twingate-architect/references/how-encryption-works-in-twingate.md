# Twingate Encryption Architecture

## Page Title
How Encryption Works in Twingate

## Summary
Twingate secures communications across four components: client-hosted Clients and Connectors, and Twingate-hosted Relays and Controller. The system achieves confidentiality and authentication without open inbound ports using TLS for Relay/Controller communications and a custom certificate-based handshake for Client-Connector traffic. Notably, even Twingate's own Relays cannot decrypt Client-Connector traffic.

## Key Information
- **Four components**: Client (end-user device), Connector (behind firewall), Relay (Twingate-hosted), Controller (Twingate-hosted)
- **No open inbound ports required** on customer infrastructure
- **Two encryption layers**: TLS for Client/Connector → Relay/Controller; session-key encryption for Client ↔ Connector
- **Controller is the root of trust** for Client-Connector authentication
- **Twingate cannot decrypt** Client-Connector traffic (end-to-end encrypted with session key)
- Connector uses a **self-signed certificate** (not CA-signed); trust is bootstrapped via Controller
- Connection Tokens are **JWT tokens signed by the Controller**
- Certificate fingerprints use **SHA-256**

## Client-Connector Authentication Flow (Step-by-Step)

1. Connector starts → generates public/private key pair → creates self-signed certificate
2. Connector sends periodic heartbeat to Controller containing SHA-256 digest of its self-signed cert
3. Client requests Connector's self-signed certificate directly from Connector
4. Client requests a Connection Token (CT) from Controller
5. Controller issues JWT (CT) containing the stored SHA-256 digest of Connector's cert → sends to Client
6. Client verifies CT authenticity, then compares SHA-256 digest in CT against Connector's cert received directly
7. Trust established → Client generates session key, encrypts it with Connector's public key, sends to Connector
8. All subsequent Client-Connector traffic encrypted with session key (symmetric)

## Configuration Values
No user-configurable encryption parameters exposed. Encryption is fully managed by Twingate.

## Gotchas
- **Connector cert is self-signed** — browsers/standard TLS would reject this, but Twingate uses Controller-mediated fingerprint verification instead of CA chain validation
- **Relays are transparent** — they forward encrypted packets but cannot inspect content; encryption applies whether path is peer-to-peer or Relay-mediated
- Session key is generated by Client, not negotiated — Client controls symmetric key material
- Controller must be reachable for Client-Connector trust establishment (it issues the JWT with the cert digest)
