# Twingate Encryption Architecture

## Page Title
How Encryption Works in Twingate

## Summary
Twingate secures communications across four components: customer-hosted Clients and Connectors, and Twingate-hosted Relays and Controller. The system achieves confidentiality and authentication using TLS for public-facing components and a Controller-rooted trust model for Client-Connector traffic. Critically, even Twingate-operated Relays cannot decrypt Client-Connector traffic.

## Key Information
- **No open inbound ports required** — Clients and Connectors initiate all connections outbound
- **Four components**: Client (end-user device), Connector (behind firewall), Relay (Twingate-hosted), Controller (Twingate-hosted)
- **Two encryption domains**: Client/Connector ↔ Relay/Controller uses standard TLS; Client ↔ Connector uses a custom trust chain
- **Controller is the root of trust** for Client-Connector authentication
- **End-to-end encryption**: Relays cannot decrypt Client-Connector packets; session key is only shared between Client and Connector
- Twingate uses Let's Encrypt as its CA for Controller/Relay certificates

## Client-Connector Authentication Flow (Step-by-Step)
1. Connector starts → generates a self-signed public/private key pair
2. Connector sends SHA-256 digest of its self-signed certificate to Controller via heartbeat
3. Client requests the Connector's self-signed certificate directly from Connector
4. Client requests a **Connection Token (CT)** from the Controller
5. Controller issues a JWT (signed by Controller) containing the SHA-256 digest of the Connector's cert
6. Client verifies CT authenticity, then compares digest in CT against digest received directly from Connector
7. On match, Client encrypts a **session key** using Connector's public key and shares it
8. All subsequent Client-Connector traffic encrypted with symmetric session key

## Configuration Values
- **Certificate digest algorithm**: SHA-256
- **Connection Token format**: JWT signed by Controller
- **Connector key generation**: Occurs at startup (self-signed certificate)

## Gotchas
- The self-signed certificate used between Client and Connector is **not** CA-signed — trust is established through the Controller's JWT, not a public CA
- If the Controller is compromised, the root of trust is broken — Controller authenticity relies on standard TLS/CA validation
- Session key is generated by the **Client**, not the Connector — the Connector only receives it encrypted under its public key
- This encryption model applies **identically** whether the transport is peer-to-peer or via Relay
