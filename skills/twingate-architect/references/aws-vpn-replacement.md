## Page Title
How to Replace the AWS VPN

## Summary
Beginner-friendly walkthrough for replacing AWS Client VPN with Twingate by deploying a Connector on an EC2 instance and creating Resources for private VPC services. No public IPs required on protected resources. Covers the full setup flow: Remote Network, Connector deploy, Resource creation, and Client installation.

## Key Information
- **Goal**: Replace AWS Client VPN with a single-Connector Twingate deployment accessible from any device
- **Resources accessed via private IP**: EC2 instances and services do not need public IP addresses
- **One Connector per VPC subnet**: Grants access to all resources on that subnet; add more Connectors for additional networks or redundancy
- **Multi-cloud**: Same approach works across AWS, GCP, Azure, and on-prem -- one Twingate network spans all

## Prerequisites
- An EC2 instance (any major Linux distro) to host the Connector
- Twingate account (Starter plan is free)

## Step-by-Step
1. Create a Remote Network in the Admin Console (name it e.g., "AWS")
2. Click the auto-generated Connector, select Linux deployment, generate tokens
3. SSH to EC2 instance, paste the generated shell command to install the Connector
4. Verify Connector status is green in Admin Console
5. Add a Resource: use private IP address or CIDR of target resources
6. Install Twingate Client on user device, join network, authenticate
7. Access resources via private IP -- verify no access when Client is disconnected

## Configuration Values
- Connector install: one-line shell command generated by Admin Console (contains access tokens)
- Resource address: private IP (e.g., `10.0.1.5`) or CIDR range

## Gotchas
- EC2 instance hosting the Connector needs outbound internet access (via IGW or NAT) to reach Twingate Controller and Relays
- Resources defined by private IP are only accessible at that IP; no DNS resolution without a private DNS Resource also defined
